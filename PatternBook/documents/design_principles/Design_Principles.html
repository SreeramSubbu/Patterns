Design Principles
Good Design are reusable,extensable and maintainable.

Encapsulate
Isolate what varies from what stays same.

- Strategy pattern

Composition
Favour Composition over Inheritence.

- Use Inheritence to achive type matching but not to get the behaviour, Ex
decorator pattern
- Composition and Delegation can be used to add behaviours at run time.

- Strategy pattern

Interface
Program to interface not to implementation.

- Strategy pattern

Loosely Coupled
Strive for loosely coupled designs between objects that interact.

- Observer pattern

Open/Closed
Classes should be open for extension to add new behaviours, but closed for
modification.

- Closing down every part of the code might be wasteful
- Using the principle might create higher level of abstraction which adds
complexity to our code.
- Goal is to design code that is resilent to change yet flexible to meet new
requirements.
- Use the principle at the areas which is likely to change.

It's a principle for object oriented design first described by Bertrand Meyer
that says that “software entities (classes, modules, functions, etc.) should be
open for extension, but closed for modification”

- Decorator pattern

Dependency Inversion
DI : Dependency Inversion principleThe principle states:
A. High-level modules should not depend on low-level modules. Both should depend
on abstractions.
B. Abstractions should not depend on details. Details should depend on
abstractions.

Depend on abstractions not upon concreate classes.
- No variable should hold reference to a concreate class
- No class should derive from a concreate class
- No method should override an implementation of a base class.

Impossible to follow, so just strive to follow.

- factory pattern

Least Knowledge
Talk only to your friends.
- Calling methods from a class that is friend (class composes) our class
creates tight coupling.
- More dependency => high coupling => less flexible => high maintainence cost.

Invoke methods that belong to
- Self
- Parameter to a method
- Any component object
- object that is cretead/instantiated
- Dont invoke methods from objects that was returned from a method call.

- This principle is equivallent to Law of Demeter.
- Heavy usage of this principle leads to high complexity and decreased
performance.

- Adapter Pattern

Single Responsibility / High Cohesion
A class should have only one reason to change

-Composite pattern

Hollywood
Don't call us , we will call you. Hollywood principle which guides us to put
decision making in high-level modules that can decide when and how to call low
level modules.
- template method pattern.

DRY
Do Not repeat yourself both code and effort.

don't repeat yourself (DRY) is a principle of software development aimed at
reducing repetition of all kinds. The DRY principle is stated as "Every piece of
knowledge must have a single, unambiguous, authoritative representation within a
system".

YAGNI
You are not going to need it : Post pone complexity , implement it only when you
have no choice

You aren't gonna need it" (acronym: YAGNI) is a principle of extreme programming
(XP) that states a programmer should not add functionality until deemed
necessary. XP co-founder Ron Jeffries has written: "Always implement things when
you actually need them, never when you just foresee that you need them."

LSP
Liskov substitution principle if S is a subtype of T, then objects of type T may
be replaced with objects of type S.
Do not use inheritance unless there is a need for subsitutability

ISP
The interface-segregation principle (ISP) states that no client should be forced
to depend on methods it does not use.
