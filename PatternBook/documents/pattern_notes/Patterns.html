Design Patterns

Pattern is a solution to a problem in a context.


Patterns allow some part of the system vary independently of all other parts.

Behavioural
Patterns that are concerned about how classes and objects interact and
distribute resposibility.

  • Strategy
    Strategy Pattern defines a family of algorithm, encapsulates each one and makes
    them interchangable. Strategy lets algorithm vary independantly of client that
    uses it.
  • Oberserver
    Observer Pattern defines a 1 to * dependency between objects so that when one
    object changes state all its dependent objects are notified and updated
    automatically.
    - Java has an in built Observer pattern support in its util package.
    - Observer can either push or pull the data from Observable, However pull is
    considered more correct.
    - Pattern is heavily used in GUI frameworks like SWING, Java Beans, RMI etc
  • Command
    Encapsulates a request as an object , thereby letting you parameterize clients
    with different requests,queue or log requests and support undo operations.
    - Use when you need to decouple an object making requests from those which
    knows how to perform it
    - Macro Commands are sets of Command objects that can be invoked, and undo-ed.
    - Command pattern can be used for queuing commands, wait for first command to
    complete execution and then execute the next one.

    - Command pattern can be used to restore to a check point. i,e Log what all
    happend to a stack, and undo it if you want to restore.
  • State Pattern
    State pattern allows an object to alter its behaviour whent its internal state
    changes. The object will appear to change its class.

    - Allows object to have differnet behaviour based on an internal state.
    - State pattern allows a context to change its behaviour according to state of
    the context
    -Strategy pattern which has the same class diagram as state diagram, configures
    /composes the context with a behaviour/algorithm
    - State pattern will typically introduce more classes into the code
  • Template
    Template method pattern defines a skelton of algorithm in a method deferring
    some steps to subclasses. It lets subclasses re define certain steps of the
    algorithm without chaging the actual structure.

    - Factory method is a specialized template method.
    - Strategy and Template pattern encapsulates algorithm one does by composition.
    Other does by inheritence.
  • Iterator
    Provides a way to access elements of a collection without exposing its
    underlying representation.
  • Mediator
  • Visitor
  • Memento
  • Chain of resposibility
  • Interpreter

Structural
Classes that lets you compose the classes or object into large structures

  • Decorator
    Decorator Patter attaches additional resposibilities to the object dynamically.
    Decorators provide flexibile alternative to subclassing for extending
    functionality.

    - Decorators have same super type as the objects they decorate
    - Decorators add its own behavior either before or after delegating to do the
    object it decorates
    - Decorator classes are typically created with builder/factory pattern.
    - Overuse leads to many small objects resulting a complex code
  • Adapter
    Converts one interface of a class to another interfce that clients expect. Helps
    classes to work together that couldn't otherwise because of incompatable
    interfaces.

    They are of two kinds, object adapters which uses composition of adaptee class
    and class adapters which uses inhertence of adaptee class.

    Class adapters requires multiple inhertence as it needs to inherit Client
    interface and Adaptee.

    Adapters converts the interface where as decorators add new responsibility.
  • Facade
    Facade provides a unified interface to a set of interface in a subsystem. Facade
    defines a higher level interface that makes the subsystem easier to use.
  • Proxy
    Proxy pattern provides a surrogate or placeholder for another object to control
    access to it.
    - Remote Proxy to control access to remote object
    - Virtual proxy to control access to resource that is expensive to create
    - Protection proxy to access a resource based on access rights.
  • Composite
    Allows to compose objects in tree structure to represent part/whole hierrachy.
    Use it when you want to treat individual and collections uniformly.
  • Bridge
    The bridge pattern is a design pattern used in software engineering that is
    meant to "decouple an abstraction from its implementation so that the two can
    vary independently"

    - Seperation of Concerns, For example in case of a webpage, allows views to
    vary independently of model.
    - UML diagram in shape of Bridge
    - Adapter makes things work after they're designed; Bridge makes them work
    before they are
  • Flyweight

Creational
Provides a way to decouple client from the objects that it needs to instantiate

  • Factory Method
    Factory Method pattern defines an interface for creating an object, but lets
    subclasses decide which class to instantiate. Factory method lets a class to
    defer instantiation to subclasses.

    To use the factory method, create a subclass of the interface an use the
    concreate subclass method to create the objects.
  • Abstract Factory Pattern
    Abstract factory pattern provides an interface for creating families of related
    or dependend objects without specifying their concreate classes.

    To use the the factory, instatiate one an pass it to a code that is written
    against the abstract type.

    F : use classes to create objects
    A : uses objects to create objects
    F : uses inheritence
    A : uses composition
    A : use this when you have families of product and you want to make sure client
    create products that belong together
    F : use this when you don't know ahead of time what all concreate classes you
    need.
  • Singleton
    Singleton Pattern ensures a class has only one instance and provides a global
    point of access to it.

    Three methods
    1. Lazy instantiantion
    - Have issue of duplication in multithreaded env. Solution 2 or 3
    2. Eager instantiation
    - Instantiate during class loading. Issue is that in case there are
    multiple class loaders duplication can occur.
    3. Double check locking
    - uses volatile static variable and sychronized creation.
  • Builder
  • Prototype

Compound
Compound pattern combines two or more patterns into a solution that solves
recurring or general problem.

- MVC is a compund pattern that consist of Observer,strategy and Composite
pattern.
-- Model makes use of observer pattern, keeping observers updated yet
decoupled.
-- View uses composite pattern to implement nested component
-- Controller uses strategy pattern, differnt strategy for different behaviour.

-- Apdapter pattern can be used ot adapt new model to an existing view and
controller.

-- Model 2 is an adaptation of MVC for web apps.
