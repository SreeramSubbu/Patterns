Design Patterns

Patterns allow some part of the system vary independently of all other parts.

Behavioural

  • Strategy
    Strategy Pattern defines a family of algorithm, encapsulates each one and makes
    them interchangable. Strategy lets algorithm vary independantly of client that
    uses it.
  • Oberserver
    Observer Pattern defines a 1 to * dependency between objects so that when one
    object changes state all its dependent objects are notified and updated
    automatically.
    - Java has an in built Observer pattern support in its util package.
    - Observer can either push or pull the data from Observable, However pull is
    considered more correct.
    - Pattern is heavily used in GUI frameworks like SWING, Java Beans, RMI etc

Structural

  • Decorator
    Decorator Patter attaches additional resposibilities to the object dynamically.
    Decorators provide flexibile alternative to subclassing for extending
    functionality.

    - Decorators have same super type as the objects they decorate
    - Decorators add its own behavior either before or after delegating to do the
    object it decorates
    - Decorator classes are typically created with builder/factory pattern.
    - Overuse leads to many small objects resulting a complex code

Creational

  • Factory Method
    Factory Method pattern defines an interface for creating an object, but lets
    subclasses decide which class to instantiate. Factory method lets a class to
    defer instantiation to subclasses.

    To use the factory method, create a subclass of the interface an use the
    concreate subclass method to create the objects.
  • Abstract Factory Pattern
    Abstract factory pattern provides an interface for creating families of related
    or dependend objects without specifying their concreate classes.

    To use the the factory, instatiate one an pass it to a code that is written
    against the abstract type.

    F : use classes to create objects
    A : uses objects to create objects
    F : uses inheritence
    A : uses composition
    A : use this when you have families of product and you want to make sure client
    create products that belong together
    F : use this when you don't know ahead of time what all concreate classes you
    need.

Design Principles
Good Design are reusable,extensable and maintainable.

Encapsulate
Isolate what varies from what stays same.

Composition
Favour Composition over Inheritence.

- Use Inheritence to achive type matching but not to get the behaviour, Ex
decorator pattern
- Composition and Delegation can be used to add behaviours at run time.

Interface
Program to interface not to implementation.

Loosely Coupled
Strive for loosely coupled designs between objects that interact.

Open/Closed
Classes should be open for extension to add new behaviours, but closed for
modification.

- Closing down every part of the code might be wasteful
- Using the principle might create higher level of abstraction which adds
complexity to our code.
- Goal is to design code that is resilent to change yet flexible to meet new
requirements.
- Use the principle at the areas which is likely to change.

Dependency Inversion
Depend on abstractions not upon concreate classes.
- No variable should hold reference to a concreate class
- No class should derive from a concreate class
- No method should override an implementation of a base class.

Impossible to follow, so just strive to follow.
